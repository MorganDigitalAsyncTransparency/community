# Project Instructions

These instructions apply to all work within discourse-observer.

## Architecture

- **Do not collapse layers.** The separation between `discourse/`, `observer/`, `model/`, `config/`, and `storage/` is intentional. Each module has a defined responsibility described in its README. Do not merge modules or move logic across boundaries without recording the decision as an ADR.

- **Keep modules focused.** Each module should do one thing well. If a module starts accumulating unrelated responsibilities, split it. If a function does not clearly belong to a module, check the module READMEs before placing it.

- **Avoid overengineering.** Do not add abstractions, configuration options, or extension points for hypothetical future needs. Build what is needed now, and extend when the need is real and understood.

## Scope

- **Preserve the generic starter direction.** This project is a reusable starting point for observing any single Discourse forum. Do not introduce assumptions about specific forums, communities, categories, tags, or workflows unless they are documented and isolated in `src/config/`.

- **Do not assume multi-forum runtime support.** Each deployment targets one forum. Do not add tenant isolation, per-forum routing, or cross-forum data structures. If multi-forum support is ever needed, it will be a separate architectural decision with its own ADR.

## Code quality

- Keep functions small and composable
- Prefer pure transformations over side effects
- Do not introduce frameworks or libraries without a documented reason
- Write tests for transformation logic and observer behavior
- Update documentation when changing module boundaries or architecture
