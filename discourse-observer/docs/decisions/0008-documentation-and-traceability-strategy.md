# 8. Documentation and Traceability Strategy

**Status:** Accepted
**Date:** 2026-03-13

## Context

The project uses test-driven development with requirements as the first step: specifications are written before tests, tests before implementation. The delivery workflow (CLAUDE.md) requires that documentation and implementation stay aligned — but without a defined traceability mechanism, there is no way to verify that alignment.

The project expects 4–5 parallel work streams, often driven by separate AI agents. Each stream works on a small number of files. Without a deliberate file organization strategy, parallel streams will create merge conflicts in shared documentation files, slowing down delivery.

The existing `specs/` directory contains behavioral specifications at varying levels of granularity. Some describe entire modules (`observer-behavior.md`), others describe system-wide constraints (`single-forum-scope.md`, `operational-constraints.md`), others define the domain model (`discourse-source-model.md`), and others define cross-cutting requirements (`use-cases.md`). These do not map one-to-one to source files, and the current structure has no defined relationship to `tests/` or `src/`.

## Alternatives Considered

### Feature-based organization

Organize `specs/` and `tests/` by feature (e.g., `specs/features/topic-snapshot/`, `specs/features/change-detection/`). Each feature gets its own directory containing requirements, design notes, and acceptance criteria.

Rejected because features cut across module boundaries, making it harder to isolate parallel work and creating cross-cutting file ownership.

### Requirement ID system

Assign a stable identifier to each requirement (e.g., `REQ-OBS-001`). Tests and implementation reference these IDs in comments or naming conventions. A traceability matrix is generated by grepping for ID patterns across the repository.

Rejected because filenames provide the same traceability without the maintenance burden of minting and managing IDs.

### Strict one-to-one file mirroring

Organize `specs/`, `tests/`, and `src/` with identical directory and file structure. Every specification file, test file, and source file share the same name in their respective directories.

Rejected because it is too rigid for real development flow — specs are written before knowing how many source files the implementation will require, and source files need freedom to evolve without restructuring specs and tests in lockstep.

## Decision

Adopt **spec-test pairing with free source structure**.

The traceability problem is separated into two concerns:

1. **Spec ↔ test:** tight pairing by naming convention. A spec describes a responsibility. Test files use the spec filename as a prefix, allowing multiple test types (unit, integration, manual) per spec. Every spec must have a defined verification method.
2. **Code → spec:** source files reference their spec and tests in a header comment. The code structure is free to evolve independently.

This accepts that specifications describe *responsibilities*, not files. A responsibility may be implemented by one source file or several. The spec-test pair is the stable anchor; the code proves itself by making the tests pass.

The full strategy — directory layout, naming conventions, verification requirements, spec lifecycle, traceability chain, CI checks, workflow, and publishing approach — is maintained in a separate strategy document at `docs/documentation-strategy.md`. That document is the authoritative reference for how the strategy is applied; this ADR records why the approach was chosen.

## Consequences

**Positive:**

- Every spec has a defined verification method — no unverified requirements
- Spec-test pairing is simple and verifiable through naming conventions
- Source code is free to evolve without restructuring specs or tests
- Parallel work streams touch different files by default, minimizing merge conflicts
- CI can verify the full chain: spec→test pairing and code→spec references

**Negative:**

- Separating tests from source files deviates from Go convention — necessary for TDD workflow, but requires explicit test configuration in Makefile and CI
- Header comments in source files will drift — accepted as technical debt managed through CI checks
- Manual verification documents depend on human discipline and may become stale
